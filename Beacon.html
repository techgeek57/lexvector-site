The Problem

County government can't use Google Analytics.

Most government websites quietly rely on Google Analytics or similar commercial tracking services to understand how residents use their digital services. These tools work — but they create real problems for government: resident data flows to third-party servers, PII exposure risks are difficult to fully mitigate, and FedRAMP compliance requirements make commercial analytics platforms a square peg in a round hole.

Grand Traverse County needed to know how residents were using gtcountymi.gov and, critically, how they were engaging with CORA Connect — the county's public AI service agent. Without analytics, there was no way to measure whether the AI was actually helping people, which pages were driving the most questions, or where residents were dropping off.

The answer couldn't be Google Analytics. It had to be something the county owned, controlled, and operated — with zero resident PII, no third-party data flows, and architecture that could pass a FedRAMP audit without exceptions.

"You can't improve what you can't measure. But you also can't measure resident behavior in a way that compromises their privacy. BEACON solves both problems at once."

BEACON Design Principle — Grand Traverse County IT
The Architecture

Serverless. Zero PII.
Entirely government-owned.

BEACON is built on a fully serverless AWS GovCloud architecture — no servers to patch, no infrastructure to manage, and costs that scale precisely with actual traffic. The system collects page-level analytics only, anonymizing every visitor through a daily-rotating SHA-256 hash of IP address, user agent, and a date salt. The hash rotates every 24 hours, making it impossible to track any individual across sessions.

The data flow for every page view is clean and auditable:

01Resident loads a page on gtcountymi.gov — beacon.js fires automatically
02POST /collect sends page path, referrer, device type — no IP, no identity
03Ingest Lambda anonymizes the visitor, writes a raw event and updates 5 daily dimension counters atomically in DynamoDB
04Raw events auto-delete after 90 days via DynamoDB TTL — daily aggregates are permanent
05Staff dashboard queries /stats, renders Chart.js charts with 7D/30D/90D range toggles — M365 login required
CORA Connect Integration

Measuring AI engagement without touching a line of CORA's code.

One of BEACON's most significant capabilities is its ability to track CORA Connect resident engagement as a separate, meaningful analytics dimension. When a resident opens the CORA Connect chat widget, BEACON starts a timer. When they close it, BEACON fires a beacon hit to the /cora-connect virtual path with the actual session duration. This data appears separately in the Top Pages dashboard — distinct from homepage traffic — giving staff a real-time view of how many residents are using the AI agent and for how long.

The entire integration is handled by a MutationObserver in beacon.js v2 watching for the CORA window's CSS state. Zero changes were made to CORA Connect's code. The bounce rate on /cora-connect is always 0% — opening the AI agent is an unambiguous signal of genuine resident engagement.

Lessons Learned

What BEACON taught us about government-owned infrastructure.

CORS in GovCloud has specific requirements

The single biggest technical challenge was CORS configuration. CivicPlus sends credentialed requests — browsers block wildcard AllowOrigins with credentialed requests. The fix required both explicit origin allowlisting and AllowCredentials=true. This is not intuitive and not well-documented for GovCloud environments. Document it the first time you solve it.

Self-hosting MSAL.js removes external CDN dependency

Rather than loading MSAL.js from a Microsoft CDN, BEACON hosts it directly in S3 GovCloud alongside the dashboard. This eliminates an external dependency, satisfies data residency requirements, and ensures the dashboard works even if CDN availability fluctuates.

Atomic DynamoDB writes prevent partial data

The ingest Lambda writes raw events and updates all five daily dimension counters (pages, referrers, devices, etc.) in a single atomic operation. If any part fails, nothing is written. This design decision eliminated an entire class of data consistency bugs before they could occur in production.

MutationObserver integration requires zero changes to tracked systems

Tracking CORA Connect engagement through a CSS class observer rather than modifying CORA's code means BEACON can be updated, replaced, or removed without touching the AI agent. Analytics infrastructure should be additive and non-invasive — not tightly coupled to the systems it observes.
